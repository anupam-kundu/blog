‚úÖ Common Cache Write Techniques

1. Write-Through Cache

How it works: Writes go to both cache and the backing store (DB).

  • ‚úÖ Strong consistency between cache and DB.
  • ‚ùå Write latency equals DB write latency.

2. Write-Around Cache

How it works: Writes go only to the DB. The cache is updated only on subsequent reads.

  • ‚úÖ Reduces write load on cache.
  • ‚ùå First read after a write will be a cache miss.

3. Write-Back (Write-Behind) Cache

How it works: Writes go to the cache and are asynchronously written to the DB.

  • ‚úÖ Low-latency writes with high throughput.
  • ‚ùå Risk of data loss if cache fails before flushing.

4. Read/Write Through Hybrid

How it works: Read-through for cache misses, write-through for updates, optionally write-around for specific cases.

  • ‚úÖ Balances consistency and performance.
  • ‚ùå More complex orchestration.

‚ö†Ô∏è Common Pitfalls in Cache Write Design

  • Stale or Inconsistent Data: Improper invalidation or delayed async writes.
  • Lost Writes: Crashed write-back cache before flushing to DB.
  • Race Conditions: Concurrent operations updating DB and cache out of order.
  • Thundering Herd / Cache Stampede: Multiple concurrent cache misses cause a burst to the DB.
  • Cache Eviction and TTL Issues: TTL too short ‚Üí frequent misses; too long ‚Üí stale data.
  • High Latency from Synchronous Writes: Seen in write-through if DB is slow.
  • Inadequate Monitoring and Metrics: Not measuring hit ratio, cache latency, or write failures.

üìå Best Practices

  • Prefer write-through for consistency-critical data.
  • Use write-around for rarely-read but frequently-updated data.
  • Use write-back with caution: add durability guarantees.
  • Invalidate or update cache atomically with DB if possible.
  • Log cache failures and monitor write error rates.
  • Test failover behavior and persistence loss scenarios.
